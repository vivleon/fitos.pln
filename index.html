DipFit

피트니스 센터 운영의
완벽한 AI·IoT 통합 운영체제

깊이(Deep) 있는 데이터로 가볍게(Dip) 쓰는 웰니스 플랫폼.
결제 코어(OmniWell) + 통제형 AI 에이전트(Seeker) 기반으로 센터, 트레이너, 회원을 단일 생태계로 연결합니다.

플랫폼 코어 및 벤치마크 역공 전략

Sprintapp.pro 전수 분석 기반 차별화 (OmniWell & Seeker)

OmniWell (센터 ERP 코어)

경쟁사가 (예정)으로 미뤄둔 결제/매출/정산을 MVP 코어로 선점. 센터 고유코드와 트레이너 좌석(Seat)을 완벽 분리하여 센터 결제자의 도입 명분을 확보합니다.

Seeker (통제형 AI 에이전트)

음성 상담 전사, 위험 신호(수면 급감 등)를 자동 탐지하되, 최종 결제/처방은 코치가 승인(Human-in-the-loop)하여 품질과 규제 리스크를 제어합니다.

회원 주권 데이터 이식성

센터 귀속에서 벗어나, 회원이 센터를 옮겨도 계정을 유지합니다. 명시적 동의(범위/기간) 기반으로만 타 센터에 공유해 신뢰 락인(Lock-in)을 만듭니다.

무손실 스키마 (PostgreSQL 100% 발췌)

CREATE TABLE centers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    center_code TEXT NOT NULL UNIQUE, -- 테넌트 식별자
    name TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'active'
);
CREATE TABLE center_seats (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    center_id UUID NOT NULL REFERENCES centers(id),
    seat_code TEXT NOT NULL, -- 트레이너 하위코드
    user_id UUID REFERENCES users(id)
);
CREATE TABLE data_share_grants (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id),
    from_center_id UUID REFERENCES centers(id),
    to_center_id UUID NOT NULL REFERENCES centers(id),
    scopes consent_scope[] NOT NULL
);

PIPA (한국) 및 생체정보 제어

Sprintapp의 방침을 넘어서는 엔터프라이즈급 준수 내재화.

제23조 (별도 동의): 신체/건강 정보는 민감정보이므로 번들링 불가. 별도 동의(Separate Consent) UX 강제.
제24조 (고유식별정보): 센터 식별 시 주민/사업자번호를 노출하지 않고 UUID(CenterCode) 활용.
감사(Audit) 대시보드: 유출/오남용 가시화 대시보드를 제품 기능으로 상용화(조회 로그 제공).

Electron 데스크톱 보안 (상담 녹취용)

상담실 PC 침해를 전제로 한 데스크톱 방어벽(권한 최소화).

  • contextIsolation: true Renderer API 노출 차단
  • sandbox: true OS 시스템 자원 접근 통제
  • nodeIntegration: false Node API 금지
  • Preload.js 통신은 saveEncryptedAudio(blob) 등 제한적 Intent만 허용

OWASP LLM Top 10 기반 에이전트 가드레일

  • LLM01 (프롬프트 인젝션)
    시스템 지침 및 도구 호출 구조는 서버 런타임에만 저장, 클라이언트 노출 차단.
  • LLM02 (불안전한 출력 처리)
    AI 출력이 DB 쓰기/결제로 직결되지 않게 스키마 검증 및 Human-in-the-loop 적용.
  • LLM06 (민감정보 노출)
    LLM 전달 전 정규식으로 PII(식별정보)를 마스킹.
  • LLM08 (과도한 에이전시)
    결제, 권한부여, 데이터 공유는 AI 자동화 불가, 인간의 최종 승인 강제.
