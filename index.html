FitOS

피트니스 센터 운영의
완벽한 AI·IoT 통합 운영체제

단순 회원 관리를 넘어선 '센터 결제/정산 + 데이터 포터빌리티 + AI/IoT 가드레일' 솔루션.
Sprintapp.pro 등 기존 SaaS의 한계를 극복하고 센터, 트레이너, 회원을 단일 생태계로 연결합니다.

플랫폼 개요 및 벤치마크 확장 전략

경쟁사(sprintapp.pro 등) 분석을 통한 차별화 및 카운터 무브 전략

센터 ERP 코어 선점

경쟁사가 (예정)으로 미뤄둔 결제/매출/정산 기능을 MVP 코어로 즉시 내장하여, 센터 결제자(Owner)의 1차 도입 명분을 확보합니다.

회원 계정 기반 데이터 이식성

센터 중심의 데이터 귀속에서 벗어나, 회원 본인 계정을 통한 데이터 소유 및 '동의 기반 타 센터 공유'로 신뢰 자산을 구축합니다.

가드레일 기반 AI (Human-in-the-loop)

AI의 과도한 권한(자동 결제/수정)을 막고, 프롬프트 인젝션을 방어하며 '초안 및 추천'에 머물게 하여 의료/보안 리스크를 완벽 제어합니다.

핵심 카운터 무브 (Counter Move)

  • 경쟁사가 템플릿(온라인 챌린지 패키지)으로 판매하는 요소를 마켓플레이스 또는 코어 기능으로 소프트웨어화.
  • IoT 장치(체성분, 스마트락커)를 직접 제조하지 않고, 파트너 연동/리스 모델(Gateway)로 흡수하여 하드웨어 CAPEX를 OPEX화.
  • 모든 AI 로직은 센터의 정책 엔진 (룰, 승인, 세이프티) 아래에서만 실행되도록 통제권 확보.

CREATE TABLE center (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  center_code TEXT NOT NULL UNIQUE,
  name TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active'
);

CREATE TABLE consent_grant (
  id UUID PRIMARY KEY,
  member_profile_id UUID NOT NULL REFERENCES member_profile(id),
  center_id UUID NOT NULL REFERENCES center(id),
  scope TEXT NOT NULL, -- 'workout'|'nutrition'|'biometrics'
  status TEXT NOT NULL DEFAULT 'active'
);

CREATE TABLE audit_event (
  id UUID PRIMARY KEY,
  occurred_at TIMESTAMPTZ DEFAULT now(),
  action TEXT NOT NULL, -- 'EXPORT_DATA', 'VIEW_TRANSCRIPT'
  actor_user_id UUID,
  details_json JSONB
);

특허 및 IP 리스크 관리 전략

경쟁사 및 선행 특허가 밀집된 영역을 파악하여 구현 우선순위 및 회피 전략을 수립합니다.

  • High Risk (혼잡 영역)

    음식 사진 기반 칼로리 자동 추정, ML 기반 맞춤형 운동 추천.
    → 완전 자동화 대신 '매니저/코치 승인(Human-in-the-loop)' 프로세스로 우회.

  • Low Risk (우선 구현)

    멀티테넌트 센터운영, 동의 기반 데이터 이식(공유), 보안 감사 및 접속 로깅.
    → IT 거버넌스 중심으로 플랫폼 차별화 동력 확보.

PIPA (한국) 및 생체정보 가이드라인

Sprintapp의 방침(HealthKit 제한 등)을 넘어서는 엔터프라이즈급 준수 내재화.

별도 동의 체계: 신체/건강 정보는 민감정보로 분류되므로 번들링 금지, 별도 동의(Separate Consent) 및 철회권 강제 보장.
논리적 분리 보관: 결제 정보, 오디오 원본, 생체 정보는 암호화 키 분리 및 접근 제어 적용.
감사(Audit) 대시보드: '유출/오남용 가시화 대시보드'를 센터장 및 회원에게 제공하여 투명성 입증.

OWASP LLM Top 10 & NIST AI RMF

  • 1. 프롬프트 경계 (Prompt-Boundary)
    시스템 지침 및 도구 호출 구조는 서버 런타임에만 저장, 클라이언트 노출 완벽 차단.
  • 2. 권한의 최소화 (Tool Permissioning)
    LLM 에이전트가 호출 가능한 도구를 센터 역할(RBAC) 기준으로 철저히 제약.
  • 3. 결과물 검증 (Output Validation)
    AI 생성 결과가 'DB 쓰기/결제/권한 변경'으로 직결되지 않도록 승인 워크플로우 강제.

Electron 샌드박싱 (Desktop 음성 녹취)

클라이언트 PC 침해를 전제로 한 데스크톱 방어벽 구축(권한 최소화).

  • contextIsolation: true Renderer API 노출 차단
  • sandbox: true OS 시스템 자원 접근 통제
  • nodeIntegration: false Node API 금지
  • IPC 통신은 제한된 의도(Intent) 기반 명령어(ex: saveRecording)만 허용
